## ❌ 未覆盖的功能测试

### 1. 远程模式测试

#### 1.1 发送消息，远程端收到回复
- **优先级**: 高
- **描述**: 测试远程 RPC 模式下的消息发送和接收
- **测试场景**:
  - 客户端通过 RPC 发送消息到服务端
  - 服务端处理消息
  - 客户端收到回复
- **前置条件**: 需要搭建 RPC 服务端
- **预估工作量**: 4-6小时

---

### 2. 状态监听细化测试

#### 2.1 思考中状态监听
- **优先级**: 中
- **描述**: 测试 Agent 进入思考状态时，客户端能收到状态更新
- **测试场景**:
  - 发送复杂问题
  - 监听到 `processing` 状态
  - 验证状态持续时间
- **测试要点**:
  - 状态变化：idle → processing → streaming → idle
  - 状态持续时间监控
- **预估工作量**: 2-3小时

#### 2.2 技能调用状态监听
- **优先级**: 高
- **描述**: 测试工具调用时，客户端收到技能调用消息
- **测试场景**:
  - 启用工具（如文件读取工具）
  - 发送需要工具调用的消息
  - 监听 `toolCallStart` 和 `toolCallResult` 事件
- **测试要点**:
  - 监听到工具调用开始事件
  - 监听到工具调用结果事件
  - 验证工具调用参数和结果
- **前置条件**: 启用内置工具或自定义工具
- **预估工作量**: 3-4小时

#### 2.3 权限申请状态监听
- **优先级**: 高
- **描述**: 测试工具权限申请时，客户端收到权限申请消息
- **测试场景**:
  - 启用需要权限的工具（如文件写入、命令执行）
  - 发送需要权限工具的消息
  - 监听 `toolPermissionRequest` 事件
- **测试要点**:
  - 监听到权限申请事件
  - 验证权限申请信息（工具名、参数等）
  - 测试权限授予/拒绝流程
- **前置条件**: 配置权限管理器
- **预估工作量**: 4-5小时

#### 2.4 回复中状态监听
- **优先级**: 中
- **描述**: 测试流式输出时，客户端收到 `streaming` 状态更新
- **测试场景**:
  - 发送消息
  - 监听状态变化：idle → processing → streaming → idle
  - 验证 streaming 状态期间的流式输出
- **测试要点**:
  - 状态转换顺序正确
  - streaming 状态期间有流式数据输出
- **预估工作量**: 2-3小时

---

### 3. 删除和清空增强测试

#### 3.1 删除处理中的消息
- **优先级**: 高
- **描述**: 消息在被处理中时，要求删除，应该打断处理后删除
- **测试场景**:
  - 发送消息
  - 在处理过程中调用 `revokeMessage`
  - 验证处理被中断
  - 验证消息被删除
- **测试要点**:
  - 处理被正确中断
  - 消息从队列和内存中删除
  - 状态恢复到 idle
- **预估工作量**: 3-4小时

#### 3.2 清空处理中的会话
- **优先级**: 高
- **描述**: 有消息被处理中时，要求清空，应该打断后清空所有消息
- **测试场景**:
  - 发送消息
  - 在处理过程中调用 `clearCurrentSession`
  - 验证处理被中断
  - 验证所有消息被清空
- **测试要点**:
  - 处理被正确中断
  - 所有消息被清空
  - 状态恢复到 idle
- **预估工作量**: 2-3小时

---

### 4. 会话状态查询

#### 4.1 客户端重连后查询会话状态
- **优先级**: 高
- **描述**: 客户端短暂离线后再次连接，需要查询当前会话状态
- **测试场景**:
  - 发送消息并开始处理
  - 模拟客户端断开连接
  - 客户端重新连接
  - 查询当前会话状态
  - 验证能正确恢复状态
- **测试要点**:
  - 支持主动查询状态（不只是监听）
  - 状态查询结果正确
  - 能正确恢复上下文
- **前置条件**: 需要状态持久化支持
- **预估工作量**: 4-5小时

---

### 5. 消息接收状态管理

#### 5.1 消息已接收状态
- **优先级**: 高
- **描述**: 标记消息已被客户端接收，避免重复推送
- **测试场景**:
  - 发送消息
  - 客户端查询消息列表
  - 标记消息为已接收
  - 再次查询时不应返回已接收的消息
- **测试要点**:
  - 消息接收状态正确标记
  - 已接收消息不会重复返回
  - 支持批量标记
- **预估工作量**: 3-4小时

#### 5.2 消息状态更新后移除已接收标记
- **优先级**: 高
- **描述**: 消息状态更新后，需要移除已接收标记，让客户端能查询到最新状态
- **测试场景**:
  - 标记消息为已接收
  - 消息状态更新（如从 sending → sent）
  - 验证已接收标记被移除
  - 客户端能查询到更新后的状态
- **测试要点**:
  - 状态更新自动清除已接收标记
  - 客户端能获取最新状态
- **预估工作量**: 2-3小时

---

### 6. 消息重发机制

#### 6.1 发送失败后重发
- **优先级**: 高
- **描述**: 客户端发送失败后，支持重发机制
- **测试场景**:
  - 发送消息失败（模拟网络错误）
  - 客户端重发相同消息ID的消息
  - 验证服务端处理逻辑
- **测试要点**:
  - 消息ID相同的情况下，不重复处理
  - 只更新消息状态
  - 返回正确的处理结果
- **预估工作量**: 4-5小时

#### 6.2 重发已处理的消息
- **优先级**: 高
- **描述**: 重发已被处理的消息，只更新状态而不重新处理
- **测试场景**:
  - 发送消息并完成处理
  - 使用相同ID重发消息
  - 验证不会重新处理
  - 验证返回正确状态
- **测试要点**:
  - 通过ID判断消息是否已处理
  - 已处理消息直接返回状态
  - 不产生重复消息
- **预估工作量**: 3-4小时

---

### 7. 打断机制

#### 7.1 任务执行中断断
- **优先级**: 高
- **描述**: 客户端发送任务后正在执行中，要求停止，需要启动 Sub Agent 判断是否终止
- **测试场景**:
  - 发送需要长时间处理的任务
  - 在执行过程中调用中断
  - 验证 Sub Agent 被触发
  - 验证任务被正确终止
- **测试要点**:
  - Sub Agent 正确判断是否应该终止
  - 任务被优雅地终止
  - 状态正确更新
  - 资源被正确清理
- **前置条件**: 实现 Sub Agent 机制
- **预估工作量**: 8-10小时

#### 7.2 权限申请状态中断断
- **优先级**: 高
- **描述**: 权限申请状态下的消息也要支持被打断
- **测试场景**:
  - 触发工具权限申请
  - 在等待权限时调用中断
  - 验证权限申请被取消
  - 验证任务停止
- **测试要点**:
  - 权限申请状态正确取消
  - 任务被停止
  - 状态恢复到 idle
  - 权限请求被清理
- **预估工作量**: 4-5小时

## 🎯 建议的测试实施顺序

### 第一阶段：核心功能补全（高优先级）

1. **技能调用状态监听** - 验证工具系统工作正常
2. **权限申请状态监听** - 验证权限管理系统
3. **删除处理中的消息** - 确保打断机制基础功能
4. **清空处理中的会话** - 确保打断机制基础功能
5. **消息已接收状态** - 避免消息重复推送
6. **消息状态更新管理** - 确保客户端能获取最新状态

### 第二阶段：重发和恢复（高优先级）

7. **发送失败重发** - 提高系统健壮性
8. **重发已处理消息** - 确保幂等性
9. **客户端重连状态查询** - 支持断线重连

### 第三阶段：高级功能（高优先级）

10. **任务执行中断断** - 需要先实现 Sub Agent
11. **权限申请中断断** - 依赖打断机制

### 第四阶段：远程模式和细化测试（中优先级）

12. **远程模式测试** - 需要搭建 RPC 服务端
13. **思考中状态监听** - 状态监听细化
14. **回复中状态监听** - 状态监听细化

## ⚠️ 注意事项

1. **测试隔离**: 每个测试应该独立，不依赖其他测试的状态
2. **超时设置**: 所有异步测试都应该设置合理的超时时间
3. **资源清理**: 测试结束后要清理所有资源（消息、会话等）
4. **并发测试**: 部分测试可能需要并发执行，注意线程安全
5. **Mock 外部依赖**: LLM 调用等外部依赖应该可以 Mock
